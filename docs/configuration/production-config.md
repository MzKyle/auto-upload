# 生产部署建议

## 推荐配置基线

| 项目 | 建议值 | 说明 |
| --- | --- | --- |
| 扫描间隔 | `30-60s` | 降低对磁盘的持续扫描压力 |
| 稳定性检查 | `3 x 5s` 或更高 | 确保采集程序已经写完 |
| 任务并发 | `2-5` | 结合 CPU、磁盘和网络压测 |
| 单任务文件并发 | `4-8` | 小文件多时可适当提高 |
| 全局上传并发 | `10-50` | 避免对 OSS 和现场网络造成冲击 |
| 上传窗口 | 网络空闲时段 | 如 `20:30-06:00` |
| 自动清理 | 谨慎开启 | 先确认 OSS 归档可靠 |

## 扫描目录布局

建议采用清晰的批次目录：

```text
/data/collector_upload/
  2026-04-29/
    batch_001/
    batch_002/
```

如果配置 `/data/collector_upload/2026-04-29` 为扫描目录，则 `batch_001` 和 `batch_002` 会分别成为任务。

## 远程机器策略

对于生产采集机，优先推荐 rsync 模式：

- 支持中断后保留部分文件
- 可以控制带宽和远端进程优先级
- 本地落盘后进入普通 OSS 上传链路
- 大文件可走分片上传

SFTP 直传适合文件较小、无需本地副本的临时同步，不建议用于超大单文件批量传输。

## 日志和留档

建议保留：

- 应用日志
- SQLite 数据库备份
- 上传失败任务的 `process_task.json`
- 远程同步失败时的机器配置和错误输出

当现场出现“OSS 上缺数据”时，优先从任务状态、`task_files`、`process_task.json` 三处交叉确认。

## 安全建议

- 使用专用 OSS RAM 用户，不使用主账号 AK/SK。
- 限制 AK/SK 只访问指定 Bucket 或 Prefix。
- 密钥文件权限仅限当前用户读取。
- 远程机器密码认证只用于测试或临时环境，生产优先密钥认证。
- 不把包含真实密钥的数据库或日志发到公开环境。

## 上线验收

上线前至少验证：

- OSS 连接测试成功。
- 小文件和大文件都能上传。
- 时间窗口符合预期。
- 任务暂停、恢复、失败重试可用。
- 新增目录能被自动扫描。
- 采集中目录不会被提前上传。
- 远程机器同步不会影响采集进程。
- 自动清理开启后只删除预期目录。
